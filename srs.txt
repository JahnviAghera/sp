1.	Introduction
1.1.	Purpose
The purpose of the SpeakSpace platform is to provide a robust, scalable, and interactive web-based environment that enables users to practice and enhance their communication skills, particularly in the context of group discussions (GD) and interviews. The system is designed to simulate real-world discussion scenarios through real-time, voice-based interaction, allowing users to engage in structured conversations with peers. A key objective of the platform is to integrate both AI-driven and human moderation to guide discussions, enforce discipline in speaking turns, and provide meaningful, data-driven feedback. By leveraging artificial intelligence, the system aims to analyze user behavior, communication patterns, and sentiment, thereby offering personalized insights that help users improve clarity, confidence, and articulation. Overall, SpeakSpace seeks to bridge the gap between theoretical preparation and practical communication experience by offering a dynamic and feedback-oriented learning environment.

1.2.	Scope
SpeakSpace encompasses a wide range of functionalities centered around real-time communication, intelligent moderation, and performance evaluation. The platform enables users to join or create live voice-based discussion rooms where multiple participants can engage in structured conversations. It incorporates AI-powered moderation features that can generate discussion topics, monitor interactions, analyze sentiment, and provide both real-time and post-session feedback. Additionally, the system supports structured speaking mechanisms such as turn-based participation, ensuring fair and organized discussions. Beyond real-time interaction, SpeakSpace also provides performance tracking and analytics, allowing users to review their progress over time through detailed insights such as speaking time distribution, sentiment trends, and improvement suggestions. The platform is designed to foster a community-driven learning environment where users can continuously practice, receive feedback, and improve collaboratively. The primary target users include students preparing for academic or placement-related group discussions, job seekers aiming to improve interview communication skills, and professionals seeking to refine their interpersonal and presentation abilities.

1.3.	Definitions & Acronyms
In the context of this document, several technical terms and acronyms are used to describe system components and functionalities. A Group Discussion (GD) refers to a structured conversation among multiple participants on a given topic, often used in academic and recruitment processes. Real-Time Communication (RTC) denotes technologies that enable instantaneous exchange of audio, video, or data between users over the internet. An AI Moderator refers to an automated system powered by artificial intelligence that manages discussions, generates topics, analyzes user interactions, and provides feedback. WebRTC (Web Real-Time Communication) is a technology that facilitates peer-to-peer audio streaming directly within web browsers without requiring additional plugins. JSON Web Token (JWT) is a secure method used for transmitting authentication data between client and server, enabling stateless session management. OAuth is an authorization framework that allows users to securely log in using third-party services such as Google, eliminating the need for separate account credentials within the platform.

1.4.	Overview
This Software Requirements Specification (SRS) document provides a comprehensive  of the functional and non-functional requirements, system architecture, and external interfaces for the SpeakSpace platform. It outlines how the system will be designed, developed, and deployed using modern web technologies, including a React.js-based frontend for building a responsive and dynamic user interface, a Node.js and Express.js backend for handling business logic and API services, and a MongoDB database for storing user, session, and interaction data. Real-time communication capabilities will be implemented using WebRTC for low-latency audio streaming, supplemented by Socket.io for managing real-time events such as user presence, speaking turns, and messaging. The document serves as a blueprint for developers, designers, and stakeholders, ensuring a shared understanding of system requirements and guiding the development process toward building a scalable, secure, and user-centric application.


2.	Overall 
2.1.	Product Perspective
SpeakSpace is designed as a modern, scalable, client-server web application that follows a modular and service-oriented architecture. The system consists of a Single Page Application (SPA) frontend developed using React.js, which ensures a highly responsive and seamless user experience without frequent page reloads. This frontend communicates with a backend system built using Node.js and Express.js, which exposes RESTful APIs for handling core business logic such as user management, room creation, session tracking, and analytics.
In addition to traditional request-response communication, the platform incorporates WebSocket-based real-time services using Socket.io to enable instantaneous bidirectional communication between clients and the server. This is essential for features such as live room updates, speaking turn management, chat messaging, and notifications. A dedicated real-time communication layer powered by WebRTC is responsible for handling low-latency, peer-to-peer audio streaming between participants in a discussion room, ensuring a smooth and natural conversation experience.
Furthermore, SpeakSpace integrates an AI processing layer that leverages external AI services (such as OpenAI APIs) to perform advanced functionalities including topic generation, natural language processing, sentiment analysis, and feedback generation. This layer operates asynchronously, processing user speech data (converted to text) and returning insights that enhance the overall learning experience. Together, these components form a cohesive system where real-time communication, intelligent moderation, and scalable backend services work in harmony to deliver a production-grade application.

2.2.	User Classes
SpeakSpace is designed to support multiple categories of users, each with distinct roles, permissions, and interaction patterns within the system.
The primary user class is the Student, who represents the core audience of the platform. Students use SpeakSpace to join or create group discussion rooms, participate in live voice-based conversations, and receive structured feedback on their communication performance. Their interaction with the system is frequent and feature-rich, involving activities such as raising hands, speaking in turns, engaging in chat, and reviewing analytics after sessions.
The second user class is the Moderator, who can either be a human facilitator or an AI-driven system. Human moderators are responsible for managing the flow of discussions, assigning speaking turns, maintaining discipline, and ensuring productive engagement among participants. They have elevated permissions that allow them to mute users, remove disruptive participants, and control session dynamics. Moderators typically possess strong communication and leadership skills, enabling them to guide discussions effectively.
The third user class is the admin, who oversees the entire platform from a system-level perspective. Admin users are responsible for managing user accounts, monitoring active sessions, handling reports, or abuse cases, and ensuring the overall health and performance of the platform. They have access to administrative dashboards and tools that provide insights into system usage, user activity, and operational metrics. Admins play a critical role in maintaining platform integrity and enforcing policies.

2.3.	User Characteristics
The users of SpeakSpace are expected to have basic digital literacy, including familiarity with web applications, account registration, and navigation through user interfaces. Since the platform heavily relies on voice-based interaction, users should be comfortable using microphones and participating in live audio conversations. A stable internet connection and access to a device with audio capabilities are essential prerequisites for effective participation.
Students, who form the majority of the user base, are typically beginners or intermediate-level communicators who are actively seeking to improve their speaking, listening, and interpersonal skills. They may include college students preparing for placements, job seekers practicing interview scenarios, or individuals aiming to build confidence in public speaking.
Moderators, on the other hand, are expected to have advanced communication skills and a good understanding of group discussion dynamics. They should be capable of managing multiple participants, resolving conflicts, and maintaining structured conversations. Their role requires attentiveness, decision-making ability, and the capacity to provide constructive feedback.
Admins are generally technical or operational personnel who are familiar with system management tools and dashboards. They should be capable of interpreting analytics, managing user data, and ensuring compliance with platform policies. While they may not directly participate in discussions, their role is crucial in maintaining a smooth and secure user experience.

2.4.	Assumptions & Dependencies
The successful operation of SpeakSpace is based on several assumptions and external dependencies that must be satisfied. Firstly, it is assumed that users will have access to a stable and sufficiently fast internet connection, as real-time audio communication and WebRTC-based streaming are highly sensitive to network latency and bandwidth fluctuations. Poor connectivity may result in degraded audio quality, delays, or disconnections, which can impact the overall user experience.
Secondly, the platform assumes that users will access the application through modern web browsers that support WebRTC technology, such as Google Chrome, Mozilla Firefox, or Microsoft Edge. Compatibility with these browsers is essential for enabling peer-to-peer audio communication without requiring additional plugins or software installations.
Another critical dependency is the availability and reliability of external AI services, particularly those used for natural language processing, sentiment analysis, and feedback generation. The system relies on APIs such as OpenAI to deliver intelligent moderation features, and any downtime or latency in these services may affect the responsiveness of AI-driven functionalities.
Additionally, the platform depends on third-party authentication services, specifically Google OAuth, to provide users with a secure and convenient login option. The availability and proper configuration of these services are essential for seamless user authentication and onboarding.
Finally, it is assumed that backend infrastructure, including servers, databases, and real-time communication services, will be deployed in a scalable and reliable environment (such as cloud platforms) to handle concurrent users and ensure high availability. These dependencies collectively define the operational boundaries within which SpeakSpace is expected to function effectively.

7.	System Architecture
7.1.	High-Level Architecture
SpeakSpace follows a distributed client-server architecture designed for scalability, modularity, and real-time performance. The system is composed of multiple interconnected layers, each responsible for specific functionalities.
The React Client acts as the presentation layer, delivering a responsive Single Page Application (SPA) that handles user interactions, UI rendering, and communication with backend services. It interacts with backend APIs over HTTPS and maintains persistent WebSocket connections for real-time updates.
The Node.js API Server functions as the core backend service, handling business logic, authentication, data processing, and RESTful API endpoints. It communicates with the MongoDB database for persistent data storage and retrieval.
A dedicated WebSocket Server (implemented using Socket.io) is responsible for managing real-time bidirectional communication between clients. It handles events such as user joining/leaving rooms, speaking turn updates, chat messages, and AI feedback delivery.
The AI Service Layer operates as a separate logical component responsible for processing user-generated content. It integrates with external AI APIs to perform tasks such as topic generation, speech analysis, sentiment detection, and feedback generation.
Finally, MongoDB serves as the primary database, storing user profiles, room data, session records, messages, and analytics in a flexible, scalable document-oriented format.
7.2.	Real-Time Communication Flow
The real-time communication flow defines how users connect and interact within a discussion room using WebSocket and WebRTC technologies.
When a user joins a room, the client first establishes a WebSocket connection with the server to synchronize room state and participant data. Through this connection, signaling messages required for WebRTC are exchanged.
Once signaling is complete, a WebRTC peer-to-peer connection is established between participants. This enables direct audio streaming without routing media through the server, reducing latency and improving performance.
After the connection is established, audio streams are exchanged in real time, allowing users to communicate seamlessly. The WebSocket layer continues to manage control events such as speaking turns, chat messages, and participant updates, ensuring synchronization across all clients.
7.3.	AI Processing Pipeline
The AI processing pipeline is responsible for analyzing user communication and generating intelligent feedback during and after discussion sessions.
Initially, audio input from users is captured and converted into text using a Speech-to-Text (STT) service. This textual data is then processed through natural language processing (NLP) models to extract meaningful insights such as sentiment, tone, and relevance.
Based on this analysis, the system generates contextual feedback, identifies communication patterns, and evaluates performance metrics. The processed results are then sent back to users via WebSocket in real time or stored for post-session summaries and analytics.
This pipeline ensures continuous monitoring and evaluation of user interactions, enabling a data-driven learning experience.

9.	API Design

9.1.	 Authentication APIs
These APIs handle user authentication, registration, and session management.
•	POST /api/auth/register: Registers a new user using email and password. 
•	POST /api/auth/login: Authenticates user credentials and returns a JWT token. 
•	GET /api/auth/google: Initiates Google OAuth authentication flow. 
9.2.	Room APIs
These APIs manage the lifecycle of discussion rooms.
•	POST /api/rooms/create: Creates a new GD room with specified configurations. 
•	GET /api/rooms: Retrieves a list of available rooms. 
•	POST /api/rooms/join: Allows a user to join a room using a code or link. 
9.3.	User APIs
These APIs handle user profile management and data retrieval.
•	GET /api/users/profile: Fetches the authenticated user’s profile information. 
•	PUT /api/users/update: Updates user profile details. 
9.4.	Analytics APIs
These APIs provide performance insights and leaderboard data.
•	GET /api/analytics/user: Retrieves individual user performance analytics. 
•	GET /api/analytics/leaderboard: Retrieves ranked leaderboard data. 
9.5.	WebSocket Events
The system uses WebSocket events for real-time communication and synchronization.
•	join_room: Triggered when a user joins a room. 
•	leave_room: Triggered when a user exits a room. 
•	speaking_turn: Manages speaking turn assignment and updates. 
•	chat_message: Handles sending and receiving chat messages. 
•	ai_feedback: Delivers AI-generated feedback to users. 
10.	Use Cases 
UC-1: Join GD Room
This use case describes how a user joins an active discussion room and begins participating.
The user logs into the platform and navigates to the list of available rooms. Upon selecting a room or entering a valid access code, the system verifies permissions and allows entry. Once inside, the user establishes real-time connections and can listen to ongoing discussions or request to speak.
UC-2: AI Moderation
This use case describes how the AI system moderates and enhances a discussion session.
At the start of a session, the AI generates or assigns a discussion topic. During the session, it continuously monitors user interactions, analyzes speech patterns, and provides real-time feedback. After the session ends, it generates a structured summary and performance insights for each participant.
UC-3: Speaking Turn Management
This use case defines how structured speaking is managed within a discussion.
A user requests to speak using the raise-hand feature. The request is added to a queue and reviewed by the moderator (AI or human). When approved, the user is granted a speaking turn and allowed to unmute and contribute. After the allotted time, the turn ends and control passes to the next participant


11.	Sequence Flows
11.1.	Joining a Room
The user initiates a request to join a room via the frontend. The request is processed by the API server, which validates access and returns room details. A WebSocket connection is then established for real-time communication. Subsequently, WebRTC signaling occurs, enabling peer-to-peer audio connections. Once completed, the user successfully joins the room and begins participation.

11.2.	AI Moderation Flow
User audio is captured and processed through a Speech-to-Text service. The resulting text is analyzed using NLP techniques to evaluate sentiment, clarity, and engagement. Based on this analysis, the AI generates feedback and insights, which are transmitted to users either in real time or after the session concludes.

11.3.	Speaking Turn Flow
A user raises a request to speak, which is added to a queue managed by the system. The moderator reviews and assigns the speaking turn. The user is then granted permission to speak for a fixed duration. Once the time expires or the user finishes, the turn ends, and the system proceeds to the next participant in the queue.






12.	Constraints
12.1.	Technical Constraints
The system relies heavily on WebRTC technology, which requires compatibility with modern web browsers. Users accessing the platform through unsupported or outdated browsers may experience limited functionality. Additionally, real-time communication performance is highly dependent on network quality, and unstable connections may lead to degraded user experience, including latency, packet loss, or disconnections.

12.2.	Legal and Privacy Constraints
SpeakSpace must comply with applicable data protection and privacy regulations, including GDPR-like standards where applicable. User data, including audio recordings and performance analytics, must be handled securely and transparently.
If session recording is enabled, explicit user consent must be obtained before recording begins. The system must also provide mechanisms for users to access, manage, or delete their personal data in compliance with privacy requirements.
